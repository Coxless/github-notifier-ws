# github-notifier-ws — コンセプト仕様書

**バージョン:** 0.1（コンセプト）
**対象プラットフォーム:** Windows 10 / 11
**形態:** システムトレイ常駐アプリ
**作成日:** 2026-06-24

---

## 1. 概要

github-notifier-ws は、GitHub の通知をデスクトップのトースト通知として届ける、Windows 常駐型のユーティリティである。一般的な通知クライアントと異なり、独自の受信箱 UI を持たない。代わりに、ユーザーが YAML ファイルに書いた**ルール**で通知の洪水を濾し、本当に必要なものだけを静かにトーストとして提示することに特化する。

設計の一語は「**静かに、濾して、必要な分だけ**」。常設の UI はトレイアイコン一点に絞り、設定はすべてファイルで行う。CLI・設定ファイル文化に慣れた開発者にとって、宣言的に通知ルールを書けることが本製品の中心的価値である。

---

## 2. 背景と競合

GitHub のデスクトップ通知アプリはすでに複数存在する。主要なものと、それぞれの性格を以下に示す。

| 製品 | 形態 | 特徴 | 設定方法 |
|------|------|------|----------|
| Gitify | メニューバー常駐・OSS | マルチフォージ（GitHub / Enterprise / Gitea 等）、フィルタ・通知音 | GUI |
| GitLight | デスクトップ・OSS | GitHub / GitLab 両対応、リポジトリ・PR・Issue 単位のフィルタ | GUI |
| DevHub | デスクトップ・OSS | TweetDeck 風のカラム UI、プッシュ通知 | GUI |
| GitHub Desktop（公式） | デスクトップ | 選択中リポジトリのイベントに限定、Action Center に残らない | — |

これらはいずれも**一覧 UI を持つ「アプリ」型で、フィルタや通知設定は GUI で行う**。設定ファイル（YAML 等）で宣言的にルールを記述する方式は見当たらない。

したがって本製品の差別化は明確である。**「軽量・トースト通知のみ・YAML 駆動のフィルタルール」**という組み合わせは既存のどれとも重ならず、設定ファイル駆動の常駐デーモンという独自のポジションを取る。

---

## 3. ゴールと非ゴール

### ゴール

- GitHub の通知を、ユーザー定義のルールで絞り込んでトースト表示する
- 通知ルールを YAML で宣言的に記述・編集できる
- トースト／トレイから最小の操作（開く・既読化）を完結できる
- 常駐しても軽量で、邪魔をしない

### 非ゴール（少なくとも初期は対象外）

- 通知の一覧 UI／受信箱ビューの提供（GitHub 本体に委譲する）
- GitHub 以外のフォージ対応（GitLab 等）
- スレッドへの返信・コメントなどの双方向操作
- モバイル対応

---

## 4. 想定ユーザー

- 一日に多数の GitHub 通知を受け取り、ノイズに埋もれている開発者
- 設定を GUI ではなくファイルで管理したい層（dotfiles・CLI 文化に親しんだ層）
- 「自分宛の mention とレビュー依頼だけ即座に気づき、CI ノイズは自動で消したい」といった、明確なフィルタ意図を持つユーザー

---

## 5. コア・コンセプトと設計原則

1. **濾すことが製品である。** 何を見せるかではなく、何を見せないかを設計する。
2. **設定はファイル。** YAML が唯一の真実。GUI の設定画面は持たない。
3. **静かに失敗する、ただし黙らない。** 設定が壊れても直前の有効な設定で動き続け、エラーは明示的に伝える。勝手に全通知を止めたりデフォルトに戻したりしない。
4. **破壊的操作は明示的に。** GitHub 側の状態を変える操作（既読化・受信箱からの削除）は既定で無効。設定で明示的に有効化したときだけ動く。
5. **常設 UI は一点。** トレイアイコンに状態を集約し、それ以外の常設ウィンドウを持たない。

## 6. 機能仕様

### 6.1 ポーリング

GitHub Notifications API（`GET /notifications`）を定期ポーリングする。以下の契約を厳守する。

- 毎回 `If-Modified-Since` に直前の `Last-Modified` を送る。変化がなければ **304 Not Modified** が返り、レート制限を消費しない。
- レスポンスの `X-Poll-Interval`（秒）を**ポーリング間隔の下限**として尊重する。設定値がこれより短くても、サーバー指定が優先される。
- 認証は classic PAT（スコープ `notifications`、Issue/Commit の詳細取得が必要なら `repo` を追加）。

### 6.2 フィルタ可能な 4 軸

`/notifications` のレスポンス単体（追加 API 呼び出しなし）でフィルタできるのは次の 4 軸である。

| 軸 | 内容 | 例 |
|----|------|----|
| `reason` | 通知のきっかけ（any-of） | `mention`, `review_requested`, `assign`, `ci_activity`, `team_mention`, `state_change`, `security_alert`, `subscribed` ほか |
| `repository` | `owner/repo` の完全一致 | `octocat/api-server` |
| `subject_type` | 対象の種別 | `Issue`, `PullRequest`, `Commit`, `Release`, `Discussion` |
| `title_contains` | タイトルの部分一致 | `[urgent]` |

**重要な制約:** `/notifications` のレスポンスには**ラベル情報・Issue/PR の状態・誰が mention したか・コメント本文が含まれない**。`subject` にあるのは title / type / url のみ。ラベルでフィルタするには通知ごとに `subject.url` を追加で叩く必要があるため、初期バージョンの対象外とし、タイトル規約（`title_contains`）で代替する。ラベル対応は将来拡張（11 章）に置く。

### 6.3 ルールエンジン

- ルールは YAML に上から順に列挙し、**最初にマッチしたルールが勝つ（first-match-wins）**。
- どのルールにもマッチしない通知は、`default` で指定したアクションに従う。
- 空の `match` はすべてにマッチするため、末尾の catch-all ルールとして使える。

### 6.4 アクション

| アクション | 動作 | 破壊的 |
|-----------|------|--------|
| `notify` | トースト表示（基本動作） | — |
| `ignore` | 何もしない。GitHub 側は未読のまま残す | — |
| `mark_read` | GitHub 側で既読化（`PATCH /notifications/threads/{id}`） | ✓ opt-in |
| `mark_done` | 既読化＋購読解除で受信箱から除去（`DELETE …/subscription`） | ✓ opt-in |

破壊的アクション（`mark_read` / `mark_done`）は、設定の `allow_destructive: true` がない限り実際には発火せず、**安全側に倒して `notify` にフォールバック**する。「気づかないうちに通知が消えていた」を防ぐための既定である。

`notify` は将来、`notify: { sound: true, priority: high }` のようにサブオプションを生やせる構造にしておく。

### 6.5 設定スキーマ（YAML）

```yaml
poll_interval: 60          # ポーリング間隔の下限（秒）。X-Poll-Interval が優先
bundle_threshold: 3        # 1回のポーリングでこの件数以上なら束ねて1枚に
allow_destructive: false   # mark_read / mark_done を実際に発火させるか
default: ignore            # どのルールにもマッチしない通知の扱い

rules:
  - name: direct
    match:
      reason: [mention, review_requested, assign]
    action: notify

  - name: active-repo PRs
    match:
      repository: octocat/api-server
      subject_type: PullRequest
    action: notify

  - name: ci-noise
    match:
      reason: ci_activity
    action: mark_read       # allow_destructive: true のときだけ実発火

  - name: urgent-by-title
    match:
      subject_type: Issue
      title_contains: "[urgent]"
    action: notify
```

設定ファイルの既定パスは `%APPDATA%\github-notifier-ws\config.yaml`。**トークンは YAML に平文で書かせず**、Windows Credential Manager に保存する。

### 6.6 ライブリロードと last-good フォールバック

- 設定ファイルを監視し、保存時に自動でリロードする。
- パースに失敗した場合は**直前の有効な設定で動作を継続**し、トーストで「設定エラー: N 行目」を通知する。全停止や暗黙のデフォルト復帰はしない。
- リロード成功時は「設定を更新しました / N 個のルールが有効です」と控えめに通知する。

### 6.7 通知の束ね（スパム対策）

1 回のポーリングで `bundle_threshold` 件以上の新着があった場合、個別トーストを連発せず「N 件の新しい通知」という 1 枚に束ねる。閾値は設定可能。

## 7. UX 仕様

### 7.1 トレイアイコン（唯一の常設 UI）

状態を色とバッジで表現する。色そのものが意味を持つ。

| 状態 | 表現 | 意味 |
|------|------|------|
| idle | モノクロ | 未読なし |
| unread | アクセント色＋未読数バッジ | 未読あり |
| paused | グレーアウト＋残り時間 | 一時停止中 |
| error | 赤系＋「!」 | 認証切れ・設定破損など |

左クリックで `github.com/notifications` をブラウザで開く。

### 7.2 右クリックメニュー

```
github-notifier-ws
──────────────────
● 3 件の未読
  2分前に同期 · API残 4,832
──────────────────
今すぐ確認
通知を一時停止  ▸  30分 / 1時間 / 明日まで止める
──────────────────
設定ファイルを開く
設定を再読み込み
──────────────────
GitHubで開く
終了
```

「一時停止（スヌーズ）」は集中作業時の必須機能。「設定ファイルを開く」は既定エディタで YAML を開くだけ。

### 7.3 トースト

**単発トースト**に載せる情報:

- タイトル: `owner/repo`
- 本文 1 行目: `reason` を人間語に翻訳（「あなたがメンションされました」「レビュー依頼」など）＋ 種別
- 本文 2 行目: `subject.title`
- ボタン: `開く` / `既読にする` / `このリポジトリをミュート`

本体クリックでスレッドをブラウザで開く。「ミュート」はその場で YAML に `ignore` ルールを追記する。

**束ねトースト**（閾値超え）: 「N 件の新しい通知」＋内訳（メンション 3・レビュー依頼 1・CI 失敗 1 など）。ボタンは `受信箱を開く` / `すべて既読にする`。

**設定エラートースト**: 「設定エラー · N 行目」＋具体的な原因（「不明なキー `subject_kind` — `subject_type` では？」）＋「直前の有効な設定で動作を継続中」。

**リロード成功トースト**: 「設定を更新しました / N 個のルールが有効です」。

### 7.4 クリックによる遷移

トースト本体・ボタンのクリックは、すべてアプリ自身の `github-notifier-ws://` プロトコル経由で処理する。クリック → 既読化してからブラウザを開く、といったアプリ側ロジックを挟めるようにする。アプリ非起動時のクリックでも、単一インスタンス機構が URL を起動中プロセスに転送することで一貫して動作する。

## 8. 技術アーキテクチャ

### 8.1 スタック

- **言語/フレームワーク:** Rust + Tauri 2
- **プラグイン:** `single-instance`（2 つ目の起動を抑止し URL を転送）、`deep-link`（`github-notifier-ws://` スキーム登録）
- **トースト:** Windows のネイティブトースト（プロトコル起動アクション付き）
- **トレイ:** Tauri 2 のトレイ機能
- **HTTP:** reqwest、**非同期:** tokio
- **設定:** serde_yaml ＋ ファイル監視（notify クレート）
- **トークン保管:** keyring（Windows Credential Manager）

### 8.2 データフロー

```
poll /notifications ─► 差分検出 ─► ルール判定 ─► アクション
   ▲ (If-Modified-Since,                          │
   │  X-Poll-Interval を下限に)         notify ────┼─► トースト（プロトコルアクション）
   └────────── sleep ◄──────────────              │
                                       mark_read ─┤─► PATCH thread
              トーストクリック          mark_done ─┘─► DELETE subscription
                    │
        github-notifier-ws://thread/<id>?do=…
                    │
        単一インスタンスが URL を転送 ─► アクション処理 ─► ブラウザ起動 / 既読化
```

### 8.3 クリック遷移の方式（重要な設計判断）

Windows でトーストのクリックをアプリに流し込む方法は 2 通りある。

1. **生の COM アクティベータ**（`INotificationActivationCallback`）を実装する方式。最も強力だが、レジストリ登録と COM 実装を自前で書く必要がある。
2. **自前プロトコル起動**でアプリに流す方式。`github-notifier-ws://` スキームを登録し、トーストのアクション引数に `…://thread/<id>?do=open` 等を持たせる。クリックで起動された 2 つ目のインスタンスが、単一インスタンス機構経由で URL を起動中プロセスに転送する。

本製品は方式 2 を採用する。生 COM を書かずに「クリック → 既読化 → 開く」やボタン処理を実現でき、コールドスタートでも動作し、Tauri のプラグイン群と相性が良いためである。

### 8.4 Action Center 永続化の要件（コード外）

プロトコル起動トーストをポップアップ消失後も Action Center に残し、後からクリックできるようにするには、unpackaged アプリの場合、スタートメニューのショートカットに **AUMID** と**スタブ CLSID**（任意の GUID。背後に COM サーバーは持たない）を付与する必要がある。これはコードではなく**インストーラの責務**である。

- インストーラ（NSIS / MSI）がスタートメニューショートカットを AUMID＋スタブ CLSID 付きで作成する。
- 初回はインストーラ経由で一度導入してからデバッグする（ショートカットが無いとトーストが表示されず、AUMID はコードと一致が必要）。

これが無いと、トースト自体は表示されるが、消すと Action Center に残らない。

### 8.5 モジュール構成

| モジュール | 責務 |
|-----------|------|
| `main` | プラグイン配線、トレイ、ウォッチャ、ポーリングループ |
| `config` | YAML スキーマ、ロード、ライブリロード、last-good フォールバック |
| `github` | API クライアント（poll / mark_read / mark_done）、トークン取得 |
| `rules` | 4 軸での first-match-wins 判定 |
| `notify` | プロトコルアクション付きトーストの構築 |
| `actions` | `github-notifier-ws://` ディープリンクの解釈と分岐 |
| `tray` | トレイアイコンの状態とコンテキストメニュー |

## 9. 名称

リポジトリ／パッケージ名は `github-notifier-ws`（`ws` = Windows）を正式名とする。製品表示名・ブランド名は別途検討中で、「静かな通知」というコンセプトを背負う和名候補として **凪（nagi）**、トレイ常駐の見張り役を表す **家守（yamori）** などを検討している。表示名は後続フェーズで確定する。

## 10. セキュリティ・プライバシー

- GitHub トークンは YAML に平文で保存せず、Windows Credential Manager に格納する。YAML 側には参照のみを持たせる。
- 通信は GitHub API（HTTPS）に限定。第三者サーバーへの送信は行わない。
- 破壊的アクションは既定で無効（6.4 参照）。

## 11. 将来拡張（初期スコープ外）

- **ラベルによるフィルタ:** ルールにラベル条件があるときだけ `subject.url` を追加フェッチしてラベルを取得する設計。API 消費を抑えるため条件付きフェッチとする。
- **通知の見た目・音・優先度:** `notify` のサブオプション化（音、優先度、アイコン色など）。
- **マルチアカウント／Enterprise 対応。**
- **`mute_repo` の YAML 自動追記:** コメントを壊さずに `ignore` ルールを追記する（surgical edit）。

---

## 12. リスクと留意点

- **Windows トーストの実装難度:** Action Center 永続化（8.4）が最大の落とし穴。インストーラ整備を初期から計画に含める。
- **トーストクレートの成熟度:** Rust の簡易トーストクレートには未成熟なものがある。プロトコル起動が要件を満たすか、最悪は生 COM 実装に切り替える退路を確保しておく。
- **URL 変換:** `subject.url`（api.github.com）から `html_url`（github.com）への変換は種別ごとに差異がある（例: `/pulls/` → `/pull/`）。ブラウザ遷移の前に変換ロジックが必要。
- **レート制限:** `If-Modified-Since` と `X-Poll-Interval` の遵守を徹底する。

## 13. 初期マイルストーン（参考）

1. ポーリング → ルール判定 → トーストの最小経路を通す
2. `github-notifier-ws://` プロトコル起動でクリック遷移を成立させる
3. トレイ状態・スヌーズ・設定ライブリロードを実装
4. インストーラ（AUMID＋スタブ CLSID）を整備し Action Center 永続化を確認
5. 破壊的アクション（mark_read / mark_done）を opt-in で実装
6. ラベルフィルタ等の将来拡張に着手
